1. Architecture Overview

The system uses a 4-stage pipeline for every single progress note:

┌──────────────────────────────────────────────────────┐ │ PROGRESS NOTE INPUT │ └──────────────────────────┬───────────────────────────┘ ▼ ┌──────────────────────────────────────────────────────┐ │ STAGE 1 & 2: AI EXTRACTION │ │ AI reads the note → returns structured booleans │ │ e.g. { mentioned: true, has_numeric_score: false } │ │ Uses constrained JSON schema │ └──────────────────────────┬───────────────────────────┘ ▼ ┌──────────────────────────────────────────────────────┐ │ STAGE 3: DETERMINISTIC RULE ENGINE │ │ Pure JavaScript — NO AI involved │ │ evaluate() → "Complete" / "Missing" / "Vague" │ │ This is the ONLY place the verdict is decided │ └──────────────────────────┬───────────────────────────┘ ▼ ┌──────────────────────────────────────────────────────┐ │ STAGE 4: AI EXPLANATION GENERATION │ │ Only runs for Missing / Vague flags │ │ Generates nurse-friendly explanations │ └──────────────────────────────────────────────────────┘
Why this matters: The AI never decides if something is compliant. It only extracts facts. A deterministic JavaScript function makes the final call. This separation is the single most important architectural decision.

2. How I Passed the Policy into the Checker

The Core Idea

I did not dump the entire policy document into one big prompt. Instead, I decomposed the policy into a structured JavaScript object where each bullet point from the policy becomes a self-contained requirement object with three parts:

PartPurposeWho Uses It
extractionSchemaTells the AI exactly what booleans to extractAI (Stage 1)
evaluate()Deterministic rule — takes booleans, returns verdictJavaScript (Stage 3)
fieldHuman-readable label for the outputUI / Excel export

Show the Code — One Requirement

Here is a single policy requirement from policy.js:

// Policy: "Resident condition — pain level (use 0–10 scale)"
{
  id: "d1_pain",
  field: "Pain scale not documented",
  extractionSchema: {
    mentioned: {
      type: "boolean",
      description: "Is pain or lack of pain mentioned anywhere in the note?"
    },
    has_numeric_score: {
      type: "boolean",
      description: "Is there a specific numeric 0-10 pain score? e.g. '2/10'"
    }
  },
  evaluate: (data) => {
    if (!data.mentioned) return "Missing";
    if (!data.has_numeric_score) return "Missing";
    return "Complete";
  }
}

What's happening:

  1. The extractionSchema creates two separate booleans — mentioned (liberal threshold) and has_numeric_score (strict threshold).
  2. The evaluate() function is pure JavaScript. No AI involved. If pain is mentioned but there's no numeric score, that's "Missing" because the policy says "use the 0-10 scale".
  3. The AI never sees the word "Missing" or "Vague". It only returns true or false.

How the Schema Flows into the Prompt

The promptBuilder.js dynamically builds a JSON schema from the policy object:

export function buildExtractionSchema(requirements) {
  const properties = {};
  for (const req of requirements) {
    properties[req.id] = {
      type: "OBJECT",
      properties: {},
      required: []
    };
    for (const [key, prop] of Object.entries(req.extractionSchema)) {
      properties[req.id].properties[key] = {
        type: prop.type.toUpperCase(),
        description: prop.description
      };
      properties[req.id].required.push(key);
    }
  }
  return { type: "OBJECT", properties, required: Object.keys(properties) };
}

This means adding a new policy requirement is one object in policy.js. No prompt editing, no new AI logic. The system automatically picks it up.

3. Walk Through a Flag

Example: Peter Parker, Day 1 — "Care plan update unconfirmed"

The note says: "Care plan: Will update."

Stage 1 — AI Extraction

The AI receives this schema for d1_care_plan_review:

{
  "mentioned": { "description": "Is the care plan mentioned at all?" },
  "is_confirmed": { "description": "Is the care plan explicitly confirmed as already reviewed and updated? 'Will update' = false." }
}

The AI returns:

{ "mentioned": true, "is_confirmed": false }

This is correct. "Will update" is future-tense. The policy asks: "Has the care plan been reviewed — Yes or No?" The answer is not yet.

Stage 3 — Deterministic Rule Engine

evaluate: (data) => {
  if (!data.mentioned) return "Missing";     // mentioned=true → passes
  if (!data.is_confirmed) return "Vague";  // is_confirmed=false → VAGUE
  return "Complete";
}

Result: ⚠️ Vague — The nurse mentioned the care plan but didn't confirm it was actually done.

Stage 4 — Explanation Generation

The AI generates: "Policy requires the care plan to be confirmed as reviewed and updated (Yes or No). Note states 'Will update' which indicates intent but not completion."

Why "Vague" and not "Missing"?

OutcomeMeaningWhen
✅ CompleteFully documented per policyAll booleans pass
⚠️ VagueTopic mentioned but lacks clinical specificitymentioned=true but clinical confirmation is false
🚩 MissingTopic is completely absentmentioned=false or critical data absent

The "Vague" category exists because vague documentation is different from no documentation. A nurse who writes "Will update" is aware of the requirement but didn't complete the action. These are different coaching conversations.

4. Could the Checker Produce a Wrong Flag?

Honest answer: Yes, absolutely. There are two types of errors:

Type 1: False Positive (flags something that's actually fine)

How it happens: The AI misreads contextual language.

Real example: "Assisted back to wheelchair." An early version treated this as mobility documentation. But the policy asks: "Can the resident move all limbs?" Being assisted to a wheelchair doesn't confirm limb movement ability.

How I fixed it: I added explicit negative examples in the schema description:

"These do NOT count: 'assisted to wheelchair', 'complaining of hip pain'
— these describe assistance, not limb movement ability."

Type 2: False Negative (misses something that should be flagged)

Real example: "Doing much better" — an early version set clinically_confirmed = true for pain outcome on Day 3. But this is not a clinical confirmation.

How I fixed it: The dual-threshold system:

LIBERAL THRESHOLD for 'mentioned' fields:
  "seems okay" → pain_mentioned = true

STRICT THRESHOLD for clinical fields:
  "seems okay" does NOT confirm full weight-bearing
  → can_full_weight_bear = false

How Would I Catch Wrong Flags?

  1. Deterministic rule engine — The AI only extracts booleans. Even if one boolean is wrong, the damage is limited. The evaluate() function is predictable, testable, and never hallucinates.
  2. Comparison testing with known samples — Two built-in residents (John Doe = perfect, Peter Parker = messy) are run after every prompt change.
  3. Extraction Preview panel — Shows raw AI booleans alongside final flags. I can see exactly which boolean was wrong.
  4. Low temperature (0.1) — Minimizes creative interpretation. For compliance checking, the AI should be as deterministic as possible.

Provider Fallback Chain

OpenAI → GitHub Models (GPT-4o) → Groq (Llama 3.3) → Gemini (primary) → Gemini (alt)

If one hits a rate limit, it's put on a 15-second cooldown and the next provider is tried automatically.

What to Say if a Flag Looks Wrong During the Demo

"Let me open the Extraction Preview to see exactly what booleans the AI returned. If a boolean is wrong, the fix is in the field description — I'd add a more specific example to the extractionSchema. The evaluate() function stays the same because the rule logic is correct; only the AI's interpretation needs refinement."

6. Alternatives Considered and Why I Rejected Them

Alternative 1: Dump the Entire Policy Into One Prompt

Approach: Send the full policy + note to the AI and ask "What's missing?"

  • Unpredictable output — Free-text analysis, inconsistent structure
  • No separation of concerns — AI decides both what to look for AND whether it passes. If it hallucinates a requirement, there's no safety net
  • Can't unit test — With free-text output, you can't write assertions
  • Prompt fragility — Changing one sentence could change the entire output

Alternative 2: Fine-Tune a Model

Approach: Collect labelled notes, train a model.

  • Data doesn't exist — No pre-labelled dataset of aged care notes tagged against this policy
  • Policy changes break the model — Re-label and re-train for every change. My approach: change one object in policy.js
  • Overkill — Decomposition + extraction works without training data

Alternative 3: Regex / Keyword Matching

Approach: Pattern matching for clinical terms.

  • Nurses write naturally — "Complaining of sore hip" means pain is mentioned but the keyword "pain" isn't there
  • Context matters — "Will update care plan" and "Care plan updated" have opposite meanings but same keywords
  • Vague vs. Missing distinction requires understanding nuance that regex can't handle

Alternative 4: Embedding-Based Semantic Search

Approach: Embed requirements and note sentences, match by cosine similarity.

  • Similarity ≠ Compliance — "Pain: comfortable" is semantically similar to "pain" but it's not compliant (missing numeric score)
  • No boolean extraction — Embeddings say topics are related, not whether specific details are present
  • Threshold tuning nightmare — What cosine score means "compliant"?

Alternative 5: Single LLM Call Per Note

Approach: Check all 15 requirements in one prompt.

  • Context window pressure — 15 complex requirements push limits, reducing accuracy
  • Error contamination — Confusion about one requirement can contaminate others
  • I actually DO group per day — But the schema forces one object per requirement ID, preventing cross-contamination

7. Key Decisions the AI Could NOT Make for Me

1. What counts as "Vague" vs. "Missing"

The AI doesn't know the difference. I defined it: Missing = entirely absent or critical data missing. Vague = topic mentioned but lacks clinical specificity. This is encoded in each evaluate() function.

2. The dual-threshold system

The AI would naturally treat all fields the same. I explicitly designed the prompt to have two modes — liberal for mentioned fields, strict for clinical fields. This distinction doesn't exist in any default AI behaviour.

3. Whether "Assisted back to wheelchair" counts as limb movement

The AI's default answer was "yes". But the policy asks a specific clinical question. I wrote explicit negative examples in the schema description to correct this.

4. The 3-day sequential structure

  • Each note is checked independently (no cross-day context)
  • Day 3 is blocked if Day 1 is missing (can't close an incident never opened)
  • Minimum 24-hour gap between days enforced

5. The escalation logic on Day 3

The policy says: "If not stabilised, escalate." The AI initially flagged notes saying "doing much better" because it interpreted "not stabilised" literally. I flipped the logic: only flag if there are explicit unresolved clinical issues. Vague positive language is treated as trending towards stabilisation.

8. Likely Follow-Up Questions

Q: "Why not use GPT-4o for everything?"

A: Cost and rate limits. Each note requires 2 AI calls. A batch of 3 residents × 3 days = 18 calls minimum. The fallback chain means if one provider is rate-limited, the system automatically tries the next. Temperature is 0.1 across all providers for consistency.

Q: "How do you know the AI returns valid JSON?"

A: Two mechanisms: (1) Gemini's native responseSchema parameter forces valid JSON. (2) For OpenAI-compatible providers, I embed a JSON example in the system prompt, set response_format: { type: 'json_object' }, and JSON.parse() with error handling.

Q: "What if the policy changes?"

A: Add, remove, or modify objects in policy.js. No prompt engineering needed. Example — adding a new requirement:

{
  id: "d1_neuro_obs",
  field: "Neurological observations not documented",
  extractionSchema: {
    mentioned: { type: "boolean", description: "Are neuro obs mentioned?" },
    has_gcs_score: { type: "boolean", description: "Is a GCS score recorded?" }
  },
  evaluate: (data) => {
    if (!data.mentioned) return "Missing";
    if (!data.has_gcs_score) return "Vague";
    return "Complete";
  }
}

That's it. The system auto-generates the schema, prompt, evaluation, and Excel output.

Q: "How would you scale to 100 residents?"

A: 100 residents × 3 days = 300 notes × 2 calls = 600 API calls. With fallback chain and rate limit handling, this takes ~25 minutes sequentially. To speed up: add parallel processing with a concurrency limiter (e.g., 3 notes simultaneously across different providers).

Q: "Why is the rule engine in JavaScript and not in the AI?"

A: Testability and trust. If I ask the AI "Is this compliant?", I can't unit test that. If I write evaluate: (data) => data.mentioned ? "Complete" : "Missing", I can test it with { mentioned: true } and know the result will never change. The AI is unpredictable; the rule engine is not.

Q: "What was the hardest part?"

A: Getting the AI to distinguish between "the nurse mentioned the topic vaguely" and "the nurse provided the specific clinical detail the policy requires." This is the entire dual-threshold system. It took multiple iterations of prompt refinement with real notes. The breakthrough was adding concrete positive and negative examples directly in each field's description.
